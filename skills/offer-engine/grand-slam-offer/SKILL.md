# Grand Slam Offer Builder

Operationalizes the entire $100M Offers playbook (Alex Hormozi) for Corey's business stack — Klient Engine community tiers, skill packs, DFY consulting, and any offer he's building or auditing for himself or clients.

The book's whole thesis: **make an offer so good people feel stupid saying no** by making it incomparable to anything else in the market. This skill walks through every component to do that — pricing, value construction, value enhancement, and naming — and produces a finished offer ready to drop into a sales page, VSL, ad, or community tier.

## When this skill fires

Wide trigger by design. Corey lives in offer-land daily. Fire on:

- Direct asks: "write me an offer," "audit this offer," "make this irresistible," "stack the value"
- Naming asks: "what should I call this," "name this program/cohort/pack"
- Pricing asks: "what should I charge," "is this priced right," "how do I justify $X"
- Guarantee asks: "what guarantee should I offer," "how do I reverse risk on this"
- Bonus/scarcity/urgency asks: "what bonuses should I add," "how do I create urgency without being scammy"
- Implicit offer work: he's drafting a sales page, community tier description, DFY package, course landing page, or VSL script and the offer itself needs sharpening
- Diagnosis: he says something is "flat," "not converting," "feels weak," "people aren't buying" — that's almost always an offer problem before it's a traffic problem

**Don't fire** when he's working on something downstream of the offer (writing hooks for an existing dialed offer, scripting a video for an offer that's already locked, doing operations work). Hooks are a different skill. This skill is for the offer itself.

## Default behavior

Apply the **full framework end-to-end** unless he scopes it down. The book is a stack — every piece compounds the next. Don't half-do it. A full Grand Slam Offer pass produces:

1. **Market check** — is the niche viable (pain, money, targetable, growing)?
2. **Pricing posture** — premium price with reasoning, anchored against the virtuous cycle
3. **Dream outcome** — the specific, vivid end state being sold
4. **Problems list** — exhaustive, in sequence, mapped to the four value drivers
5. **Solutions list** — every problem flipped to "how to" language
6. **Delivery vehicles** — the actual *how*, run through the Delivery Cube cheat codes
7. **Trim & stack** — high-value/low-cost items kept; bundled into named sub-offers with dollar values
8. **Scarcity** — which type and how to apply honestly
9. **Urgency** — which type and how to apply honestly
10. **Bonuses** — stacked, named, with proof and price tags, eclipsing core offer value
11. **Guarantee** — chosen from the four types (or stacked), worded with teeth
12. **Name** — built from the MAGIC formula

The output is a **finished offer document**, not a lecture about offers. Lead with the deliverable.

## Workflow

### Step 1 — Intake (conversational, not a form)

Before drafting anything, get the inputs you need. Don't drown him in questions — ask only what's missing. If he's already given you the product, market, and price in the conversation, skip ahead. The minimum you need:

- **What is the product/service?** (the actual thing being sold)
- **Who is the avatar?** (the specific person, not "small business owners")
- **What's the dream outcome?** (what does the avatar want, in their words)
- **Current price (if any) and current offer (if any)** — needed for audits, optional for net-new
- **Constraint:** is this for a specific channel/format? (community tier copy, sales page, VSL, ad, DM script)

If you're missing 2+ of these, ask in one batch via natural language — not a checklist. If it's an audit and he pasted the existing offer, extract these from the paste first, then only ask about gaps.

### Step 2 — Read the relevant references

Then load the references you need. **You don't need to load all of them every time.** Route by what's being asked:

| If the task is | Read these references |

| Full offer build (default) | All seven, in order |
| Audit existing offer | `01-market-and-pricing.md` + `02-value-equation.md` + whichever enhancement layer is weakest |
| Just stack value / build the bundle | `02-value-equation.md` + `03-problems-solutions-delivery.md` + `04-trim-and-stack.md` |
| Just the guarantee | `06-guarantees.md` |

The references are dense. They contain the actual frameworks, examples, scripts, and cheat sheets pulled from the book — not summaries. Use them as you would a working reference, not a textbook.

### Step 3 — Build the offer

Work through the stack. The book is explicit that the order matters: **market → pricing posture → value construction → value enhancement → naming**. Don't jump to naming before the value is built; don't add bonuses before the core offer is dialed.

A few things to internalize before producing:

- **Solve EVERY perceived problem.** Not most. Every one. The single missing problem is often the reason a prospect doesn't buy. (See the eating-out-guide story in `03-problems-solutions-delivery.md`.) When in doubt, list more problems, not fewer.
- **Bonuses should eclipse the core offer in stated value.** This is psychological — it makes the core offer feel underpriced.
- **Guarantee teeth:** every guarantee is `If [condition], then [specific consequence].` Vague guarantees ("satisfaction guaranteed") have no teeth. Name the guarantee.
- **Premium pricing is the moral choice**, not the greedy one. The virtuous cycle of price (in `01-market-and-pricing.md`) is the argument — invoke it when Corey or his clients flinch at the number.
- **Value comes from the bottom of the value equation, not the top.** Reducing time delay and effort/sacrifice is harder and more competitive than inflating dream outcomes. Beginners over-promise; pros remove friction.

### Step 4 — Output format

Produce the offer as a **clean, structured document Corey can paste into a sales page, deck, or doc**. Default structure:

```
[OFFER NAME]
[One-line promise — the dream outcome with timeframe]

THE PROMISE
[2-4 sentences. What they get, by when, with the guarantee baked in.]

WHO THIS IS FOR / NOT FOR
[Sharpen the avatar. Repels the wrong fit, magnetizes the right one.]

WHAT YOU GET (THE STACK)
Component 1: [Named sub-offer] — [one-line outcome] — Value: $X,XXX
  • [bullet of what's inside / how it works]
  • [bullet]
Component 2: [Named sub-offer] — [one-line outcome] — Value: $X,XXX
  • ...
[repeat]

BONUSES
Bonus 1: [Named bonus] — [one-line outcome] — Value: $XXX
  [Why this exists / what specific obstacle it crushes]
Bonus 2: ...
[repeat]

TOTAL VALUE: $XX,XXX
YOUR INVESTMENT: $X,XXX

THE GUARANTEE
[Named guarantee. Conditional or unconditional. Specific teeth.]

WHY NOW (SCARCITY + URGENCY)
[The honest, real reason the window is finite. No fake countdowns.]
```

If Corey wants this in a different format (sales page copy, VSL script, community tier description, ad creative brief), adapt — but the underlying components stay the same. The structure above is the **content spine**; format is the wrapper.

If the deliverable is long enough or formal enough to warrant a file (full sales-page draft, multi-tier offer architecture, full DFY packaging doc), put it in a markdown or docx file via the relevant skill (`klient-engine-assets` if it's a KE deliverable). For a quick offer rewrite or audit, output inline in the chat.

### Step 5 — When the user wants iteration

Most offer work is iterative. After producing v1, expect Corey to push back on specific pieces — the price, the guarantee, the name, a bonus. Re-route to the relevant reference and tighten that piece without rebuilding the whole stack. Keep the rest stable so he can compare.

## What this skill specifically does for Corey's stack

Some context-specific notes since this skill lives in Corey's environment:

- **Klient Engine community tiers**: the freemium → paid → DFY funnel is itself a stack of offers. Each tier should be its own Grand Slam Offer with its own dream outcome, stack, bonuses, and guarantee. Don't write tier descriptions like feature lists — write them like offers.
- **Skill packs**: each pack is a productized offer. They should have a named promise, a clear dream outcome (what the buyer can DO after installing it), a stack of what's included with dollar values, and ideally a guarantee tied to a specific outcome (e.g., "ship your first piece of UGC in 7 days or your money back").
- **DFY consulting / equity engagements**: these are the highest-ticket offer in the stack. They warrant the most detailed Grand Slam treatment — performance-based guarantees (implied guarantees in the book) are often the right move here since they align incentives and let Corey charge a premium with a strong reverse-risk story.
- **Don't fall into hype-y Hormozi-isms.** Corey's brand is honest positioning over hype. Apply the framework's *structure* with restraint in the *language*. No "feel stupid saying no" copy in actual deliverables — that's the internal mental model, not the customer-facing tone. The output should sound like Corey: direct, confident, systems-minded, no manufactured urgency theater.
